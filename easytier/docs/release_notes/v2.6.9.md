# EasyTier v2.6.9 Release Notes

This release is a fork release focused on Stealth rollout, QUIC/KCP Proxy
readiness, KCP lifecycle cleanup, GUI status stability, underlay-loop hardening,
and Linux veth fallback validation.

## Critical Performance Warning

`secure_mode=true + stealth_mode=true` currently has a known throughput
regression on the tested TCP underlay path.

Remote validation on 2026-07-08, collected on the v2.6.8 release candidate
before the v2.6.9 version bump, showed:

| Mode | 512 MiB transfer |
| --- | ---: |
| Plain TCP underlay | `107.75 MB/s` |
| `stealth_mode=true` with runtime-derived secure | `107.91 MB/s` |
| Explicit `secure_mode.enabled=true` only | `107.83 MB/s` |
| Explicit secure + Stealth | `10.90 MB/s` |

RSS remained stable at about `15 MB`, while CPU approached one saturated core.
This looks like a CPU hot-path issue, not a memory leak. The root cause is not
confirmed yet. Operators who need high throughput should avoid enabling explicit
`secure_mode=true` together with Stealth until profiling and a fix are complete.

Runtime-derived secure for Stealth is not the same as explicit global
`secure_mode`: it protects Stealth-protected PeerConn payloads, but does not
publish RoutePeerInfo public keys and does not enable global relay/session secure
mode.

## Highlights

- Multi-transport Stealth support and capability handling for `udp`, `tcp`,
  `faketcp`, `quic`, `wg`, `ws`, and `wss`.
- Default Stealth and transport priority policy for new configurations.
- UDP Stealth compatibility fixes for fallback budget, hole-punch negotiation,
  duplicate Syn, phase transition, and listener strictness.
- QUIC/KCP Proxy prepare READY ACK, classified fallback reasons, and health
  handling.
- KCP connect lifecycle cleanup for cancelled hedged connects, stale state, and
  orphan timeout cleanup.
- Direct transport priority with RTT guard and safer failover/failback behavior.
- WAN/LAN direct candidate classification fix: public IPv4/IPv6 candidates no
  longer become LAN solely because they are on-link, and link-local candidates
  must match a local link-local network before receiving LAN priority.
- GUI status refresh stabilization, normalized mixed-case backend data, stable
  chart updates, and Proxy Failover status display.
- Underlay guard and local breaker hardening for fake-IP/TUN candidate pollution
  and EasyTier/Mihomo/sing-box loop amplification.
- Linux native veth NIC fallback under the existing `tun` feature for restricted
  environments where the TUN device is unavailable but network administration
  capabilities exist.

## Compatibility And Operational Notes

- Fixed strict UDP Stealth listeners silently drop legacy plain probes by design.
  TCP strict anti-legacy behavior has a known same-secret mixed-mode gap and is
  tracked separately.
- `transport_priority` only affects direct-connect. Proxy failover remains fixed
  as `QUIC -> KCP -> Native`.
- The v2.6.9 direct priority validation is recorded in
  [Transport Priority And Underlay Breaker Validation - 2026-07-09](../transport_priority_breaker_validation_2026_07_09.md).
  Relay rows in `peer`/`route` output can coexist with a lower-latency DIRECT
  path and should not be read as proof that data is currently using relay.
- `disable_kcp_input` and `disable_quic_input` only affect Proxy input
  capability. They do not shut down the corresponding underlay listeners.
- When using Mihomo, sing-box, Clash, NekoBox, Throne, or similar system TUN
  proxy tools, exclude EasyTier/Tailscale processes and virtual CIDRs from the
  proxy/TUN path. EasyTier's underlay guard reduces bad candidates but cannot
  replace process or route bypass in the system proxy tool.
- Docker artifacts are intentionally not part of this fork's release workflow.

## 中文发布摘要

本版本主要面向 Stealth rollout、QUIC/KCP Proxy 就绪确认、KCP 生命周期清理、GUI 状态页
稳定性、underlay 回流防护以及 Linux veth fallback。

重要警告：`secure_mode=true + stealth_mode=true` 当前在已测 TCP underlay 路径上存在
明显吞吐回归。2026-07-08 远端验证中，plain、派生 secure Stealth、单独显式 secure
均约 `107 MB/s`，但显式 secure + Stealth 约 `10.90 MB/s`。RSS 稳定，CPU 接近单核满载。
这更像 CPU 热路径问题，不像内存泄漏。高吞吐场景暂不建议默认组合开启显式
`secure_mode=true` 和 Stealth。

运行期派生 secure 不是“没加密”：它会保护 Stealth-protected PeerConn payload；但它也
不等同于显式全局 `secure_mode`，不会发布 RoutePeerInfo 公钥，也不会启用全局
relay/session secure mode。
