# EasyTier v2.6.10 Release Notes

This release adds safe default IPv6 listening for QUIC without changing public
configuration, wire format, Stealth negotiation, Proxy ordering, or other
transport listeners.

## Highlights

- An unspecified IPv4 QUIC listener with a fixed nonzero port now receives an
  optional IPv6-only companion on the same port when IPv6 is enabled.
- IPv4 and IPv6 QUIC sockets use explicit `V4Only` and `V6Only` bind modes, so
  they coexist without dual-stack overlap or port ambiguity.
- Explicit `0.0.0.0:port` plus `[::]:port` QUIC configuration is deduplicated to
  exactly one socket per address family.
- Standalone explicit `[::]:port` retains dual-stack semantics.
- QUIC validates the actual bound address family, IP, and port before publishing
  a listener. A suppressed IPv6 bind failure can no longer appear as an active
  port-zero listener.
- Failure of the optional IPv6 companion does not remove the IPv4 listener or
  alter process-global client endpoint-pool behavior.
- `quic://0.0.0.0:0` remains IPv4-only and emits a warning because two independent
  random-port binds cannot guarantee a shared firewall port.
- Hostname listeners, FakeTCP IPv6 behavior, QUIC/KCP Proxy order, and mixed-version
  wire compatibility remain unchanged.

## Validation

The exact code candidate was built by the rolling profiling beta workflow and
validated on five x86_64 Linux hosts, including Linux 3.10, IPv4-only LAN nodes,
and two public IPv6 nodes.

- Every default fixed-port QUIC listener exposed exactly one IPv4 and one IPv6
  socket on the configured port.
- A real IPv6 QUIC connection was established through the generated `[::]`
  listener with multi-transport Stealth enabled.
- Blocking IPv6 QUIC removed only `quic6`; TCP6/UDP6/IPv4 QUIC kept the overlay
  reachable. Restoring the path recreated `quic6` within approximately 8 seconds.
- An occupied IPv6 companion port produced a strict bind mismatch while the IPv4
  listener and instance remained operational.
- Explicit IPv4 plus IPv6 QUIC configuration produced one listener event and one
  socket per family, with no duplicate companion.
- LAN QUIC throughput reached approximately 670-735 Mbit/s forward and 718 Mbit/s
  reverse in the tested topology. Idle CPU remained about 0.4-1.0%, RSS about
  19-27 MiB, and no panic, bind, or listener errors remained after convergence.

## Compatibility Notes

- Enabling IPv6 expands the reachable UDP surface. Existing firewall policy and
  strict-Stealth configuration remain authoritative.
- Automatic IPv6 companion creation requires a fixed nonzero QUIC port. Users
  who require a random IPv6 port may still configure `quic://[::]:0` explicitly.
- Endpoint removal by local address is an older, separate lifecycle risk and was
  intentionally not refactored in this release.

## 中文发布摘要

本版本在不修改公开配置、wire 格式、Stealth 协商、Proxy 顺序和其他 listener 行为的
前提下，为固定非零端口的默认 QUIC listener 增加安全的 IPv6 监听。

- `quic://0.0.0.0:端口` 在启用 IPv6 时会生成同端口的 `[::]` IPv6-only companion。
- IPv4/IPv6 listener 分别使用明确的 `V4Only/V6Only`，不会与 dual-stack socket 重叠。
- 显式同时配置 `0.0.0.0` 和 `[::]` 时，每个地址族只创建一个 socket。
- QUIC 在发布 listener 前严格核对实际 IP、地址族和端口；IPv6 bind 被底层忽略后形成
  `[::]:0` 假监听的问题已被阻止。
- IPv6 companion 失败不会影响已有 IPv4 listener，也不会修改全局客户端 endpoint pool。
- 端口 `0` 不自动生成 companion，并输出明确 warning；固定防火墙端口请使用非零端口。

候选版本已在五台 Linux 主机上验证，包括 Linux 3.10、纯 IPv4 局域网和公网 IPv6。
真实 `quic6` 在开启多协议 Stealth 时成功建立；掐断 IPv6 QUIC 后其他热备保持 overlay
可用，恢复后约 8 秒内重新建立 `quic6`。局域网 QUIC 测得约 670-735 Mbit/s 正向和
718 Mbit/s 反向吞吐，收敛后 CPU、RSS、socket 数量稳定。
